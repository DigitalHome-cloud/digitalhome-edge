**Top items requiring attention this cycle**:

1. **C-1 / C-2** — Both the MCP server (`:8000`) and the Node-RED admin API (`:1880`)
   listen on the LAN with **zero authentication**, and `install.sh` explicitly opens
   both ports in `ufw`. Anyone on the LAN can operate every device, exfiltrate the
   cloud Bearer token, or edit flows / inject code. This is not "in a hostile environment
   this would matter" — this is "the box currently works this way and is deployed."
2. **H-1** — Even if MCP is fixed, Node-RED HTTP-in endpoints (`/api/lights/*`, `/api/heating/*`,
   scenes, etc.) are also unauthenticated. Add `httpNodeAuth` in `settings.js` or gate
   `/api/*` behind a per-endpoint token.
3. **H-3** — `digitalhome.edge.config.cache` file mode is inherited from the source
   (~644), leaving the cloud API key, Hue key, and Node-RED credential secret readable
   by any local user. Enforce `chmod 600` in `install.sh`.
4. **M-1** — Systemd units have no hardening (`NoNewPrivileges`, `ProtectSystem`,
   `PrivateTmp`, …). A compromised MCP server or Node-RED palette can do anything
   `dhc-svc` can.

## Findings

### CRITICAL

#### C-1. Node-RED admin API is unauthenticated and LAN-open
**Domain**: 10. Node-RED posture (edge-extension)
**Evidence**:
- `install.sh:307–309` — `ufw allow 1880/tcp comment 'Node-RED'`
- `install.sh` writes `settings.js` via `sed` to set `credentialSecret` and enable Projects
  (lines 205–222) but **never inserts an `adminAuth` block**. The default Node-RED
  settings shipped by `npm install -g node-red` has no admin auth.
- `docs/install.md:158` "settings.js — Projects enabled, credentialSecret set" — confirms
  intent to configure `settings.js` but auth is not on the checklist.

**Impact**: Any host on the LAN can browse to `http://<edge-ip>:1880`, edit any flow,
export `flows_cred.json` (which is encrypted at rest but is decrypted in-memory by
Node-RED — an attacker with admin can read credentials via the editor UI or via
`/credentials`), inject arbitrary function nodes (JavaScript runs as `dhc-svc`), or
deploy malicious flows that call outbound to the Hue Bridge, CCU, and cloud. Full RCE
as `dhc-svc` in one hop.

**Remediation**:
1. Add an `adminAuth` block in `settings.js` with a bcrypt-hashed password (or better,
   OAuth via the cloud identity provider once available). Bootstrap the credential in
   `install.sh` alongside `credentialSecret`.
2. Consider binding Node-RED to `127.0.0.1` (`--host 127.0.0.1`) if no LAN client
   currently needs the editor; access via `ssh -L 1880:localhost:1880`.
3. Optionally add `httpAdminAuth` on top of `adminAuth` for defense-in-depth on the
   Admin API itself.

#### C-2. MCP server is unauthenticated and LAN-open
**Domain**: 9. MCP surface (edge-extension)
**Evidence**:
- `mcp-server/server.py:347` — `uvicorn.run(mcp.sse_app(), host="0.0.0.0", port=8000)`
- `install.sh:310` — `ufw allow 8000/tcp comment 'MCP server'`
- No `Authorization`, no bearer check, no session token in `server.py`. The bind on
  `0.0.0.0` is intentional (docs/install.md step 5 tells operators to configure Claude
  Desktop with `http://192.168.1.10:8000/sse`).

**Impact**: Any LAN peer can:
- Call `nodered_trigger("/api/scene/away", …)` — control every device.
- Call `nodered_get_flows()` — dump the full Node-RED graph including inline function code.
- Call `nodered_inject(node_id)` on any node — arbitrary flow invocation.
- Call `cloud_post("/anything", {…})` — use the box's `cloud.api_key` Bearer token
  against `https://api.digitalhome.cloud` for the box's tenant. This bypasses the
  cloud plane's per-user auth entirely.
- Call `kb_add(…)` / `agent_log_write(…)` — poison the shared agent memory.

**Remediation**:
1. Add a shared-secret bearer check to `mcp.sse_app()`, either via a FastMCP middleware
   or by wrapping the Starlette app. Store the token in the config cache alongside the
   other secrets.
2. Prefer binding to `127.0.0.1` for the near term; only bind LAN-wide when
   token auth is enforced.
3. In the SLAB5 target architecture (`docs/architecture.md`) the MCP endpoint moves to
   `:8443` (implied TLS + streamable HTTP). Add TLS at that migration; do not defer
   authentication until then.

### HIGH

#### H-1. Node-RED HTTP-in endpoints have no authentication
**Domain**: 10. Node-RED posture
**Evidence**: `flows/digitalhome-flows/flows.json` — 19 `http in` nodes across
State/Lights/Scenes/Heating/Cloud Sync/Devices tabs. None carry any auth wrapper.
`settings.js` has no `httpNodeAuth` block.

**Impact**: A LAN attacker who doesn't want to route through the MCP tools can hit
`POST http://<edge>:1880/api/lights/kitchen/on` directly. Same effect as C-2 but with
a lower dependency surface.

**Remediation**: Add `httpNodeAuth: { user: "dhcedge", pass: "$2b$..." }` in
`settings.js`; require Basic auth on `/api/*`. Coordinate with the MCP server so its
`httpx` calls include the header (read the credential from config cache). Alternative:
front all `/api/*` with a small middleware node that checks a shared token header.

#### H-2. MCP tool signatures accept arbitrary paths (SSRF surface for authenticated callers)
**Domain**: 9. MCP surface
**Evidence**: `mcp-server/server.py:120–156` — `nodered_trigger(endpoint, payload)`,
`nodered_query(endpoint)`, `nodered_inject(node_id)`, `cloud_get(path)`,
`cloud_post(path, payload)`, `cloud_patch(path, payload)` all pass the caller's
`endpoint` / `path` verbatim into an HTTP URL:
```
r = await c.get(f"{NODERED_URL}{endpoint}", ...)
r = await c.post(f"{CLOUD_API_URL}{path}", json=payload, headers={"Authorization": ...})
```

**Impact**: Today C-2 makes this moot (anyone can call anything). Once auth lands on
the MCP endpoint, this pattern still lets any *authenticated* caller reach:
- Any Node-RED HTTP-in path or Admin API path (e.g. `/flows`, `/credentials`).
- Any digitalhome.cloud REST path using the box's Bearer token — including tenant
  boundaries the caller may not be entitled to at the cloud plane.

**Remediation**: This resolves naturally in the SLAB5 target — the plan (Phase 3.4)
retires all `nodered_*` and `cloud_*` tools in favour of a catalog-driven dispatcher
where each Tool is a specific action bound to a specific Point. Until then:
- Add an allowlist of endpoint prefixes accepted by `nodered_trigger` /
  `nodered_query` (e.g. `/api/state/`, `/api/lights/`, `/api/heating/`).
- Restrict `cloud_*` paths similarly, or route them through a per-operation tool.

#### H-3. Config cache file is world-readable
**Domain**: 1. Secrets & credentials
**Evidence**: `install.sh:168–169`:
```
cp "${REPO_DIR}/digitalhome.edge.config.cache.example" "${CONFIG_CACHE}"
chown "${DHC_USER}:${DHC_USER}" "${CONFIG_CACHE}"
```
No `chmod` — the file inherits umask defaults (0644). The example file in the repo is
`-rw-rw-r--` (line-count check: `ls -l digitalhome.edge.config.cache.example`).
The file holds `cloud.api_key`, `nodered.credential_secret`, and
`devices.hue-bridge.api_key`.

**Impact**: Any local user (e.g. `frank-uwe` admin, guest accounts, an intruder who
lands as a lower-privileged local user) can read the cloud Bearer token, the
Node-RED credential-encryption key (which unlocks `flows_cred.json`), and the Hue
API key. Once secrets are pulled they can be exfiltrated at leisure.

**Remediation**:
1. Add `chmod 600 "${CONFIG_CACHE}"` after the `chown` in `install.sh`.
2. In the SLAB5 target (`docs/architecture.md` §6, ADR-0001), secrets move to
   `/opt/dhe/secrets/*` with 0600 perms. Bring that split forward to the current
   layout without waiting for full cutover.

#### H-4. No TLS on any LAN-facing surface
**Domain**: 9./10. MCP + Node-RED
**Evidence**: `mcp-server/server.py:347` uvicorn without TLS; `install.sh:240`
`--port 1880` no TLS; `docs/install.md:88` `http://192.168.1.10:8000/sse`
(explicitly HTTP).

**Impact**: Passive LAN sniffers see every device command, every knowledge-base
write, every cloud API call (headers included — but Bearer token itself is only sent
from box → cloud over HTTPS, so it's not leaked here; the leak is the device-control
traffic and any local response bodies).

**Remediation**: Terminate TLS at a local reverse proxy (Caddy/nginx) with a
self-signed cert or a local ACME-DNS cert, and update `docs/install.md` step 5 to use
`https://`. In the SLAB5 target, MCP already moves to `:8443` with TLS implied.

### MEDIUM

#### M-1. Systemd units have no sandboxing / hardening
**Domain**: 11. Systemd / runtime hardening (edge-extension)
**Evidence**: `install.sh:229–266` — both `nodered.service` and `mcp-server.service`
consist of `Type`, `User`, `Group`, `WorkingDirectory`, `Environment`, `ExecStart`,
`Restart`. **No** `NoNewPrivileges`, `ProtectSystem`, `ProtectHome`, `PrivateTmp`,
`PrivateDevices`, `RestrictAddressFamilies`, `CapabilityBoundingSet`,
`SystemCallFilter`, `ReadWritePaths=`.

**Impact**: A compromise of Node-RED (via C-1) or of the MCP server (via C-2 combined
with a vulnerability in FastMCP / uvicorn / a Python dep) gets full write access to
`/home/dhc-svc/`, the ability to `execve` any binary, network access to the entire
LAN + WAN, and can escalate to root via any local suid vuln in the future.

**Remediation**: Add a hardening block to both units, e.g.:
```
NoNewPrivileges=yes
ProtectSystem=strict
ProtectHome=read-only
PrivateTmp=yes
PrivateDevices=yes
RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX
CapabilityBoundingSet=
AmbientCapabilities=
ReadWritePaths=/home/dhc-svc /var/log
```
(Adjust `ReadWritePaths` per service — Node-RED needs `/home/dhc-svc/.node-red`,
MCP needs the DB and config paths only.)

#### M-2. Python dependencies unpinned; no lockfile
**Domain**: 5. Dependency hygiene
**Evidence**: `mcp-server/requirements.txt` uses `>=` for all four deps
(`mcp>=1.27.0`, `httpx>=0.28.0`, `python-dotenv>=1.2.0`, `aiosqlite>=0.21.0`). No
`requirements.lock` / `pip-tools` output / `uv.lock`. A reinstall on a new box will
resolve to whatever versions PyPI currently offers, and a compromised transitive
dependency can slip in unnoticed.

**Remediation**: Adopt `pip-tools` or `uv` to produce a locked `requirements.txt`
(fully pinned including transitives with hashes). Regenerate on planned upgrades.

#### M-3. Node-RED and palette versions drift on reinstall
**Domain**: 5. Dependency hygiene
**Evidence**: `install.sh:104` — `npm install -g --silent node-red` (no version).
`install.sh:116–117` — `"node-red-contrib-ccu": "^3.4.2"`, `"node-red-contrib-huemagic": "^4.2.2"` (caret ranges). No `package-lock.json` committed for the Node-RED userDir.

**Impact**: Two boxes provisioned two months apart end up on different Node-RED /
palette versions with different behaviour and different published CVE exposure.

**Remediation**: Pin Node-RED (`npm install -g node-red@X.Y.Z`) and commit a
`package-lock.json` for the userDir; regenerate on intentional bumps.

#### M-4. `dhcedge update-hosts` doesn't validate hostname / IP before writing `/etc/hosts`
**Domain**: 11. Systemd / runtime hardening
**Evidence**: `bin/dhcedge:93–120` — reads `hostname` and `ip` fields from the config
cache JSON via Python and interpolates them into the managed `/etc/hosts` block
without regex validation. Newlines and comment characters in JSON string values
would be honoured (JSON `"\n"` is a real newline once parsed).

**Impact**: Low today because the config cache is edited by trusted operators. Rises
to Medium/High once the SLAB5 cloud-driven config sync lands (Phase 5) — a
compromised cloud config push could inject arbitrary `/etc/hosts` entries and MITM
any subsequent DNS resolution done by `dhc-svc`.

**Remediation**: Validate `hostname` against `^[a-zA-Z0-9.-]{1,253}$` and `ip`
against a proper IPv4/IPv6 regex (or `ipaddress.ip_address(...)`) before writing.
Reject the block on any invalid entry and log the offending row.

#### M-5. Runtime duplicates the repo tree at `/home/dhc-svc/mcp-server/`
**Domain**: 6. Build & CI/CD
**Evidence**: `install.sh:140–153` — copies `server.py`, `requirements.txt`, `.env`
from `${REPO_DIR}/mcp-server/` into `/home/dhc-svc/mcp-server/`, then installs the
venv there. The systemd unit's `WorkingDirectory` (line 258) points at the copy,
not the repo. Ongoing updates require re-running the copy or manually editing the
runtime file.

**Impact**: `git pull` on the repo does not update the running server; operators
patching the "obvious" file (the repo copy) see no effect. The SLAB5 Docker
migration removes this problem entirely, but it exists on the current stack.

**Remediation**: Interim — replace the copy with a symlink (`ln -s`) or point the
systemd unit's `WorkingDirectory` at `${REPO_DIR}/mcp-server/` directly. Long term:
Phase 1 of the SLAB5 plan makes this moot by packaging the server as a container
built from the repo checkout.

### LOW

#### L-1. `README.md` is a stub
**Domain**: 8. Documentation
**Evidence**: `README.md` has 2 lines total (`# digitalhome-edge / digitalhome edge server`).

**Impact**: New contributors landing from the GitHub UI see no orientation. Given
the repo now has `CLAUDE.md`, `SPEC.md`, `docs/architecture.md`, and
`docs/adr/0001-dhe-alignment.md`, the README should at minimum point at these.

**Remediation**: Expand README with a one-paragraph "what this is" + pointers to
the four docs.

#### L-2. No test suite
**Domain**: 7. Code quality
**Evidence**: No `tests/`, no `pytest.ini`, no test targets in any script.
`mcp-server/server.py` is 348 lines with all tool logic in it.

**Impact**: Any refactor of the auth story (C-2 remediation) or the tool signatures
(H-2 remediation) can regress silently. The Phase 3 MCP-server rewrite (SLAB5 plan)
is a large diff with no safety net.

**Remediation**: Add `pytest` + a small integration harness that spins up the MCP
server against a mock Node-RED. Prioritize auth-boundary tests.

#### L-3. `ufw` opens `:8000` unconditionally
**Domain**: 11. Systemd / runtime hardening
**Evidence**: `install.sh:310` unconditional `ufw allow 8000/tcp`. Even in the
"stage" mode (services disabled), the port is opened.

**Impact**: Minor — the service isn't running in stage, but the firewall says
"come on in" which is confusing for defenders reviewing the box's exposure.

**Remediation**: Move the `ufw allow 8000/tcp` behind the `if [[ $INSTALL_MODE == "prod" ]]`
gate, or (better) only open it once the MCP endpoint has auth enabled.

#### L-4. Mixed policy on `.claude/` — gitignored but selectively force-added
**Domain**: 8. Documentation
**Evidence**: `.gitignore` now includes `.claude` (staged in this session); yet
`.claude/dhc-security-audit/SKILL.md` and `.claude/dhc-security-audit/CHECKLIST.md`
are also staged (`git add -f` semantics implied by their `A` status in `git status`).

**Impact**: The next reader sees `.claude` in `.gitignore` and reasonably assumes
nothing under it is tracked. The audit skill files being tracked is a deliberate
policy departure that isn't documented anywhere.

**Remediation**: Either (a) narrow the gitignore rule to exclude the skill dir
(`.claude/*` + `!.claude/dhc-security-audit/`), or (b) add a top-level comment in
`.gitignore` explaining the exception. Option (a) is safer against accidental
force-adds of other `.claude` contents.
