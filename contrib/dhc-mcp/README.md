# node-red-contrib-dhc-mcp

MCP (Model Context Protocol) server for Node-RED. Turns any Node-RED
flow into a set of MCP tools and resources callable by Claude or any
other MCP client.

## Why in Node-RED

The MCP server used to be a separate Python container in the
digitalhome.edge stack. That created a hop for every tool call:
Claude → Python MCP → HTTP → Node-RED → device. Consolidating into
Node-RED removes the hop, unifies auth and logging, and lets the flow
generator produce a single artifact (`flows.json`) that carries both the
tool wire and the flow logic.

See `docs/architecture.md` §9 and the ADR at `docs/adr/0001-dhe-alignment.md`
for the trade-off analysis.

## Nodes

### `mcp-server-config` (configuration node)

Singleton per Node-RED instance. Owns:

- The MCP endpoint URL registration on Node-RED's Express (SSE + POST).
- Bearer token authentication (loaded from a mounted secret file).
- The C-BOX loader — parses `cbox.jsonld`, generates `tools/list` +
  `resources/list`, watches the file for changes and emits
  `notifications/tools/list_changed`.
- The session and callId registry — per SSE connection, tracks pending
  responses.

Every `mcp-tool-in`, `mcp-resource-in`, and `mcp-response` node
references this config node.

### `mcp-tool-in` (inbound)

Fires when a `tools/call` request arrives for a specific tool name.
Emits `msg` = `{ payload: <arguments>, mcp: { callId, sessionId, toolName } }`.
Wire into the device-facing flow.

### `mcp-resource-in` (inbound)

Fires on `resources/read` for a specific resource URI.
Emits `msg` = `{ payload: null, mcp: { callId, sessionId, uri } }`.
Wire into the state-reading flow.

### `mcp-response` (outbound)

Terminates a flow. Reads `msg.payload` (tool result or resource
content) and `msg.mcp.callId` (must round-trip from the `-in` node),
and sends the response to the originating MCP client.

## Wire structure

One triplet per Point in the C-BOX (auto-generated by the Phase 3 flow
generator, or hand-authored during development):

```
[mcp-tool-in: "livingroom_lights_on"]
    → [function: validate args]
    → [ccu: dimmer command]
    → [function: wrap result]
    → [mcp-response]
```

## Configuration

The config node reads a JSON blob at startup (path defaults to
`/config/dhe.config.cache`, overridable via env var
`DHC_MCP_CONFIG`). Expected fields:

```json
{
  "mcp": {
    "path":       "/mcp",
    "auth_token_file": "/secrets/mcp-auth-token",
    "cbox_path":  "/cbox/cbox.jsonld",
    "server_info": { "name": "digitalhome-edge", "version": "0.3.0" }
  }
}
```

Endpoints exposed on the Node-RED Express instance:

- `GET  /mcp/sse` — SSE stream, requires `Authorization: Bearer …`
- `POST /mcp/messages/{sessionId}` — client → server messages
- `GET  /mcp/health` — unauth healthcheck (returns `{ ok: true }`)

## Development

```bash
cd contrib/dhc-mcp
npm install
npm test
```

To install into a local Node-RED for manual testing:

```bash
cd ~/.node-red
npm install <path-to>/contrib/dhc-mcp
```

Restart Node-RED. Palette shows four new nodes under the "digitalhome"
category.
